TASK 6 — CONFLICTING TRANSACTIONS · N:N CONTENTION DEMO
Two real MySQL transactions fire simultaneously in separate Python threads, both attempting to sign the same driver to different teams.

Transaction A fires first: runs SELECT DriverID … FOR UPDATE, acquiring an exclusive row-level lock on the DRIVER row, then sleeps 1.2 seconds to simulate slow processing while holding the lock.

Transaction B fires 100ms later: hits the same SELECT … FOR UPDATE — and blocks waiting for TXN A to release the lock.

When TXN A COMMITs, the lock releases. TXN B then proceeds — but our business logic detects the driver was already re-signed by TXN A and issues a ROLLBACK.

How It Works: MySQL Lock Contention
SELECT … FOR UPDATE
Acquires an exclusive row-level lock on the matched rows. Any other transaction trying to read or update those rows with FOR UPDATE will block until the first transaction commits or rolls back. This is MySQL's built-in serialization mechanism — it prevents dirty reads and lost updates in concurrent writes.
InnoDB MVCC
InnoDB uses Multiversion Concurrency Control — readers don't block writers and writers don't block readers for snapshot reads. But FOR UPDATE explicitly opts into locking. Combined with the REPEATABLE READ isolation level used here, each transaction sees a consistent snapshot from the moment it began, even as other transactions commit.
